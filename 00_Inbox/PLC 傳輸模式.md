## 1. Modbus RTU (Remote Terminal Unit)

這是工業界最常見的模式，特別是在 RS-485 或 RS-232 串列通訊中。

- **資料格式：** 二進制（Binary）。資料直接以 8-bit 字節傳輸，效率最高。
- **偵錯機制：** 使用 **CRC (Cyclic Redundancy Check)** 循環冗餘檢查碼，確保資料傳輸完整性。
- **同步方式：** 依靠「時間」來區隔封包。標準規定，封包與封包之間必須有至少 **3.5 個字元的停頓時間（Silence）**，接收端才會認為一個完整的封包已結束。
- **應用場景：** 變頻器控制、感測器讀取、PLC 之間的串列通訊。

## 2. Modbus ASCII

這是一種較老舊、效率較低，但對人類閱讀與除錯較友善的模式。

- **資料格式：** 使用 ASCII 字元。例如，要傳輸十六進制 `0x4A`，RTU 只傳 1 個 Byte，但 ASCII 會拆成 `'4'` 和 `'A'` 兩個字元傳輸。
- **偵錯機制：** 使用 **LRC (Longitudinal Redundancy Check)** 縱向冗餘檢查碼。
- **封包特徵：** * 開始符號：冒號 `:`
    
    - 結束符號：換行符號 `CR/LF` (\r\n)
        
- **優點：** 因為有明確的開始與結束符號，對通訊時間的要求不嚴格，適合一些通訊品質較差或需要手動輸入指令測試的環境。

## 3. Modbus TCP

這是 Modbus 在乙太網路（Ethernet）上的變體，目前在智慧工廠與物聯網（IoT）中極為普及。

- **資料格式：** 二進制，封裝在 TCP/IP 封包內。
- **傳輸機制：** 預設使用 **Port 502**。
- **封包變動：** * **移除 CRC：** 因為 TCP 協議本身就具備強大的校驗機制，所以 Modbus TCP 封包尾部不需要 CRC。
    
    - **新增 MBAP Header：** 封包頭部增加了 7 個字節的「Modbus 應用協議標頭」，用來標識交易序號、協議類型與長度。
        
- **優點：** 速度極快，且可以在既有的區域網路（LAN）內跑，不需要額外拉序列線。

## 核心差異比較表

| **特性**   | **Modbus RTU** | **Modbus ASCII** | **Modbus TCP**          |
| -------- | -------------- | ---------------- | ----------------------- |
| **編碼格式** | 二進制 (Binary)   | 可讀字元 (ASCII)     | 二進制 (Binary)            |
| **物理層**  | RS-232 / 485   | RS-232 / 485     | Ethernet (RJ45 / Wi-Fi) |
| **錯誤檢查** | CRC-16         | LRC              | TCP/IP 校驗               |
| **封包結尾** | 3.5 字元靜默       | CR/LF            | 無 (由 TCP 控制)            |
| **傳輸效率** | 高              | 低                | 極高                      |
