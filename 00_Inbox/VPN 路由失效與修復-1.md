## 📌 問題現象 (Problem)

- VPN 介面 `ppp0` 狀態為 `UP`。
- 無法 `ping` 通目標主機 `192.168.68.10`。
- 檢查路徑發現封包走錯路：`ip route get` 顯示走實體網卡 `eno1` 而非 `ppp0`。

## 🔍 根本原因分析 (Root Cause)

當 VPN（L2TP/PPP）撥號連線後，系統必須在**路由表**中增加一條規則，告訴核心：「要去 192.168.68.0 網段的封包，請交給 ppp0 處理」。

本次故障是因為 **路由規則丟失**：

1. **原因**：可能是手動重啟 Server B 或是 VPN 不正常斷線，導致自動產生的靜態路由被刪除。
2. **結果**：系統找不到特定路由，於是根據「預設路由 (Default Gateway)」將封包丟往家用的實體分享器 (`192.168.31.254`)。分享器並不認識內網 VPN 網段，因此封包直接被丟棄 (Drop)。

## 🛠 修復指令與邏輯 (The Fix)

### 1. 診斷指令

```bash
# 查看封包預計路徑
ip route get 192.168.68.10
```

> **若顯示 `dev eno1` 代表路由失效。**

### 2. 手動校正

```bash
# 強制指定網段流量走 ppp0
sudo ip route add 192.168.68.0/24 dev ppp0
```

### 3. 驗證成功 (本案例結果)

```
64 bytes from 192.168.68.10: icmp_seq=1 ttl=63 time=23.0 ms
```

- **TTL=63**：代表封包經過了一跳 (Hop) 抵達目標，確認是在 VPN 網段內跳轉。
- **23ms**：代表隧道連線品質良好，足以支撐影像推論傳輸。

## 🚀 進階建議：自動化防範

為了避免每次重啟都要手動輸入，可以將路由指令加入撥號腳本：

我們用 `tee` 指令直接把內容寫進去（這需要 `sudo` 權限）：

```bash
sudo tee /etc/ppp/ip-up.d/add-virgo-route <<EOF
#!/bin/sh
# 當 ppp 介面啟動時，自動補上通往 Virgo (H200) 的路徑
# \$1 是 pppd 傳入的參數，代表介面名稱 (例如 ppp0)
ip route add 192.168.68.0/24 dev \$1
EOF
```

#### 2. 賦予執行權限

如果檔案不能執行，系統會跳過它。請執行：

```bash
sudo chmod +x /etc/ppp/ip-up.d/add-virgo-route
```
