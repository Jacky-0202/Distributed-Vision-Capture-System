## 摘要

- **節點 A (Gateway)**: `192.168.35.209` (hipoint)
- **節點 B (Bridge)**: `192.168.31.68` (tec-H610I-DDR4)
- **關鍵架構**: 跨網段 L3 路由 (TTL=63)，中繼節點 B 同時掛載 VPN。

---
## 網路環境診斷

原本兩台伺服器位於不同網段（`.35.x` 與 `.31.x`），曾嘗試手動掛載同網段 IP 但因實體隔離（ARP Incomplete）失敗。
最終確認透過內部網關 `192.168.31.254` 即可達成跨網段通訊。

### 1. 基礎連通性驗證 (Ping Test)

| **測試方向**  | **目標 IP**        | **結果** | **關鍵數據** | **結論**      |
| --------- | ---------------- | ------ | -------- | ----------- |
| **B ➔ A** | `192.168.35.209` | ✅ 成功   | TTL=63   | 路由已通，經過 1 跳 |
| **A ➔ B** | `192.168.31.68`  | ✅ 成功   | TTL=63   | 雙向路由均正常     |

語法：
```bash
ping -c 4 192.168.35.209
```

考慮防火牆：
```bash
sudo ufw allow 8000/tcp
```

---
## 關鍵修復：VPN 路由劫持 (Split Tunneling)

在 Server B 連接 VPN 後，發現無法 Ping 通 Server A。這是因為 VPN 奪取了預設閘道 (Metric 50 < 100)。

#### 🛠️ 永久修復方案 (Netplan)

為了確保重開機後連線不中斷，已將靜態路由寫入設定檔。

- **路徑**: `/etc/netplan/50-cloud-init.yaml`
- **內容**:
```yaml
network:
  version: 2
  ethernets:
    eno1:
      dhcp4: true
      routes:
        - to: 192.168.35.0/24      # 指定 Server A 網段
          via: 192.168.31.254      # 強制走實體網關
```
