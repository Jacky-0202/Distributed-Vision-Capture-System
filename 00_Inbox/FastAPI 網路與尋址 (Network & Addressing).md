## 摘要

在開發分散式系統（如 Master-Slave 架構）時，最首要解決的問題就是「設備之間如何互相看見與溝通」。
以下是從單機開發跨入區域網路 (LAN) 通訊的關鍵知識點。

---
## 1. 網路定址的三要素 (IP, Port, Path)

在 Master 呼叫 Slave 時，必須精準組合出目標的 URL：`http://<目標_IP>:<目標_Port>/<API_端點>`

- **目標 IP (如 `192.168.34.144`)：** 就像是**建築物的地址**。負責讓網路封包找到特定的那台實體機器。可以透過 `ip a` 指令查詢。
- **通訊埠 Port (如 `:8000`)：** 
	就像是建築物裡的**門號**。一台 CM4 可以跑很多服務，FastAPI 預設站在 8000 號門。
	加了 `:8000` 才能精準找到 FastAPI 程式，而不是敲到 SSH 或資料庫的門。
	
- **API 端點 Path (如 `/capture/0`)：** 進門後要執行的具體任務。

## 2. 伺服器綁定：127.0.0.1 vs 0.0.0.0

這是 Web 服務部署時最常踩的坑。

- **`127.0.0.1` (回環位址):** 只允許「住在同一台機器上的程式」互相連線。若 Slave 綁定此位址，遠端的 Master 發出請求會被視為「外人」而直接遭拒 (Connection Refused)。
- **`0.0.0.0` (任意位址):** 允許這台機器所有網卡接收來自外部的連線請求。分散式架構必備。
    
    - 啟動指令：`uvicorn main:app --host 0.0.0.0 --port 8000`
        

## 3. 連線方向性：誰需要知道誰的 IP？

在網路通訊中，**「發起請求的人 (Client)」一定要知道「接收請求的人 (Server)」的 IP。** 這決定了系統架構的設計：

- **情境 A：Master 主動去「拉 (Pull)」照片 (GET 請求)**
    
    - **角色：** Master = 發起者，Slave = 接收者。
    - **IP 需求：** Master 必須把 Slave 的 IP 寫在程式碼裡。Slave 不用知道 Master 是誰，只要開著門等就好。
        
- **情境 B：Slave 主動把照片「推 (Push)」給 Master (POST 請求)**
    
    - **角色：** Slave = 發起者，Master = 接收者。
    - **IP 需求：** Slave 拍完照後，必須知道 Master 的 IP 才能繳交照片。這時 Master 不需要知道 Slave 的 IP。

## 4. 進階考量：克服動態 IP 的變動

如果透過路由器分配的動態 IP (DHCP) 因為重開機而改變，寫死在程式碼裡的 IP 就會失效。常見的解法有兩種：

1. **設定靜態 IP (Static IP):** 在路由器或 CM4 系統內綁定固定的網址。
2. **mDNS (Multicast DNS):** 在區網內使用 `.local` 主機名稱互相呼叫（例如 `http://cm4-slave-1.local:8000`），取代容易變動的數字 IP。
