## 摘要

在開發結合實體硬體（如相機、感測器、GPIO 腳位）的 FastAPI 服務時，
硬體資源的管理邏輯與一般的資料庫連線完全不同。如果把控制硬體的程式碼隨便塞進 API 端點 (Endpoint) 裡，系統很快就會崩潰。

---
## 1. 為什麼硬體資源需要「獨立管理」？(實體設備的陷阱)

如果您在每次使用者呼叫 `/capture` 的時候，才去執行 `Picamera2()` 啟動相機，會遇到兩個致命問題：

- **啟動延遲 (Warm-up Delay)：** 軟體變數的宣告只要幾奈秒，但硬體通電、感測器初始化、鏡頭對焦、自動曝光適應環境，**通常需要 2 到 3 秒**。如果寫在 API 內部，每次請求都要等 3 秒才能拍照，這在分散式系統中會導致嚴重的 Timeout 骨牌效應。

- **資源獨佔 (Resource Locking)：** 硬體資源通常是「排他性」的（Exclusive）。如果第一個 API 請求開啟了相機還沒關閉（或因為報錯而忘記關閉），第二個 API 請求進來時，系統就會拋出 `Device or resource busy` 的底層錯誤，導致整個伺服器當機，必須重啟 CM4 才能解決。

## 2. FastAPI 的解法：生命週期管理 (`Lifespan`)

FastAPI 提供了 `@asynccontextmanager`（非同步上下文管理器），專門用來處理這種「與伺服器同生共死」的重型資源。

**它的運作邏輯是「預熱與常駐」：**

1. **伺服器啟動前 (Startup)：** 先把相機打開、對焦好、進入待命狀態。
2. **伺服器運行中 (Running)：** API 端點只負責下達「咔嚓（擷取當下那一幀畫面）」的指令，速度極快（毫秒級），不碰硬體初始化。
3. **伺服器關閉時 (Shutdown)：** 攔截關閉訊號（如按 `Ctrl+C`），確實執行 `camera.close()`，釋放實體設備的佔用。

---
## 💻 核心實作結構

```python
from fastapi import FastAPI
from contextlib import asynccontextmanager

# 全域變數，用來存放預熱好的硬體物件
hardware_camera = None 

@asynccontextmanager
async def lifespan(app: FastAPI):
    global hardware_camera
    
    # === [階段 1：啟動 (Startup)] ===
    print("啟動相機並等待曝光穩定...")
    hardware_camera = init_camera_function() 
    
    yield  # 🌟 伺服器在這裡持續運行，接受 API 請求
    
    # === [階段 2：關閉 (Shutdown)] ===
    print("收到關機訊號，正在安全釋放相機資源...")
    hardware_camera.release_hardware()

# 將 lifespan 綁定到 FastAPI 實例上
app = FastAPI(lifespan=lifespan)
```
