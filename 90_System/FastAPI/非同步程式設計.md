## 1. 核心概念：同步 (Sync) vs 非同步 (Async)

要理解非同步，我們可以用「去餐廳點餐」來做比喻：

- **同步 (Synchronous / 阻塞型 Blocking)：** 您在櫃檯點了一份炸雞。您就一直站在櫃檯前死盯著店員，什麼事都不做（卡死），直到 10 分鐘後炸雞做好交給您，您才端著盤子回座位。
    
    - _程式表現：_ 執行 `time.sleep(10)` 或傳統的 `requests.get()`。整個伺服器在這段時間內會「凍結」，無法服務其他進來的請求。
        
- **非同步 (Asynchronous / 非阻塞型 Non-blocking)：** 您在櫃檯點了炸雞，店員給您一個「震動叫號機」。您拿著叫號機回到座位上，這 10 分鐘內您可以滑手機、看書，甚至去隔壁買杯飲料。等叫號機震動了，您再去把炸雞拿回來。
    
    - _程式表現：_ 執行 `await asyncio.sleep(10)` 或 `await client.get()`。伺服器在等待的空檔，會把運算資源釋放出來（交還給 Event Loop），去處理其他 API 請求。
        

## 2. 為什麼 Master 節點「必須」用非同步？

想像一下，如果您的 Master 需要呼叫 4 台 Slave 上的相機拍照，而每台相機對焦加上網路傳輸需要 2 秒鐘。

- **如果用同步寫法 (排隊)：** 呼叫相機 A (等 2 秒) ➡️ 呼叫相機 B (等 2 秒) ➡️ 呼叫相機 C (等 2 秒) ➡️ 呼叫相機 D (等 2 秒)。 **總耗時：8 秒。** 這樣根本稱不上「分散式處理」，只是在排隊。
    
- **如果用非同步寫法 (發叫號機)：** 發送請求給 A (立刻切換) ➡️ 發送給 B (立刻切換) ➡️ 發送給 C (立刻切換) ➡️ 發送給 D。然後 Master 同時等待這 4 個設備的回應。 **總耗時：約 2.1 秒。** 這才是發揮硬體效能的完美架構！
    

## 3. 語法解析：`async def` 與 `await`

在 Python 與 FastAPI 中，這兩個關鍵字通常是成對出現的：

- **`async def` (宣告這是一個會交出控制權的函式)：** 當您在函式前面加上 `async`，您就等於告訴 Python：「這個函式裡面會有一些需要等待的任務，請把它放進 Event Loop（事件迴圈）裡排程。」
    
- **`await` (標記等待點，也就是發出叫號機的瞬間)：** 只能在 `async def` 裡面使用。它告訴程式：「這裡要等一下外部的結果（例如等網路封包、等資料庫查詢）。現在我把 CPU 控制權還給你，等結果回來了再叫醒我繼續往下跑。」

```python
import asyncio
from fastapi import FastAPI
import httpx

app = FastAPI()

# 加上 async，宣告這是一個非同步的 API 端點
@app.get("/trigger")
async def trigger_task():
    print("1. 發送請求給相機")
    
    # 加上 await，代表「這裡要花時間等網路回應，CPU 先去忙別的吧」
    async with httpx.AsyncClient() as client:
        response = await client.get("http://192.168.34.144:8000/capture")
        
    print("2. 收到照片了，繼續往下執行")
    return {"status": "done"}
```

#### ⚠️ 絕對要避免的「毒藥」：在非同步中混入阻塞代碼

這是初學者最常犯的致命錯誤。如果您宣告了 `async def`，但裡面卻用了**同步（會卡死）的套件**，那整個非同步機制的 Event Loop 就會被這個套件徹底鎖死！

- ❌ **絕對不要用：** `time.sleep()`、`requests.get()`。
- ✅ **必須替換為：** `await asyncio.sleep()`、`await httpx.AsyncClient().get()`。
