## 📑 核心哲學 (Philosophy)

- **目標**：電腦 A 的內容必須與 GitHub（電腦 B 上傳的內容）**完全同步**。
- **包含**：新增的檔案要下載、修改的要覆蓋、**刪除的檔案也要跟著刪除**。
- **策略**：放棄電腦 A 的所有舊紀錄，直接強行對齊雲端。

---

## 🚀 執行步驟 (在電腦 A 執行)

這套指令比 `git pull` 更強大、更乾淨：

### 第一步：抓取雲端最新情報

這會讓電腦 A 知道雲端（電腦 B 推送後）現在長什麼樣，但還不會更動本地檔案。

```bash
git fetch origin
```

### 第二步：強制對齊雲端 (核心指令)

這會讓電腦 A 的 `main` 分支指針直接跳到跟雲端一樣的位置。


```bash
git reset --hard origin/main
```

> [!NOTE] 為什麼不用 git pull？
> 
> `git pull` 會嘗試合併（Merge），如果兩邊歷史太亂會報錯。`reset --hard` 則是直接抹除電腦 A 的差異，達到 100% 同步。

### 第三步：清除電腦 A 多出來的雜物 (選用)

如果你在電腦 A 有一些「電腦 B 沒創過、雲端也沒有」的雜亂測試檔，這行會把它們通通刪掉，確保資料夾完全乾淨。

```
git clean -fd
```

---
## 🛠️ 為什麼 `git pull` 有時候不夠？

|**情況**|**git pull 的反應**|**reset --hard 的反應**|
|---|---|---|
|**檔案衝突**|停止執行，叫你手動修復。|**直接覆蓋**，以雲端為準。|
|**B 刪除了檔案**|通常會同步刪除。|**絕對同步刪除**。|
|**A 有多出的雜檔**|會保留那些雜檔。|**(配合 clean) 全部清空**。|
|**歷史紀錄斷層**|報錯 (Refusing to merge...)。|**無視斷層**，直接跳到最新點。|
